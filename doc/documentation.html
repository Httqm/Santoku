What is Santoku ?

Santoku is a Python script designed to generate configuration files for Shinken and Nagios from data provided in a CSV file. The CSV file is expected to have a specific format. The generated files are :

  • hosts.cfg, including host groups
  • services.cfg
  • commands.cfg
  • and a very basic tabular-presented Nagvis map showing ALL hosts listed in the CSV file (0% done)

At a previous position, I wrote a PERL script doing all this (except the commands.cfg part). This script still works very well, but because of the numerous plugins / environments / usages it handles, it's now a 60KBytes cluttered script and debugging/updating it has become a nightmare. So I can't even share it with the community.
So I started developping Santoku :

  • because I wanted to build a modular tool that others could benefit from, without being developers themselves
  • to improve my Python skills
  • ... and because coding is fun ;-)
The concept

Basically speaking, just consider a few things :

  • a CSV file you can generate with a spreadsheet programm such as (Open|Libre)Office Calc or Microsoft Excel. This file contains one host per line, and services on columns. Placing values in the cells defines the monitoring scenario.
  • the Santoku script
  • a configuration file for every plugin you're using
Why "Santoku" ?

Well, ... there was Batman and Robin. Now, there is Shinken and Santoku :-)
Actually, since Santoku is designed to be used together with Shinken, I wanted to name it as a Japanese cutting device. However, since Santoku is a very humble piece of software, I could not name it as a deadly weapon.
So here comes "santoku", which is a japanese kitchen knife.

Requirements

Santoku is developed on a Debian Squeeze (64 bit) workstation, with Python 2.6.6 and Python 3.1.3, and using CSV files made with Calc from OpenOffice.org 3.2.1 .

Some minor fixes will be necessary to support CSV files made with Microsoft Excel and to handle files made in a Microsof Windows environment :

  • remove extra quote signs
  • end of line encoding (?)
Files

Here's a short description of the files of the Santoku script, as found on GitHub :

  • santoku.py : main program file
  • deployToShinken.sh : a Bash script to automate deployment of generated files to Shinken + restart, either locally or through SSH.
  • convertConfNag2SantokuCsv.sh : before writing Santoku, I made a previous PERL script to do a similar job, but performance was "OOPS", code quality was "ARGL" and maintenance was "OMFG" (This is why I decided to develop Santoku, actually). This PERL script was called ConfNag, it already used a formatted CSV file to hold data, and convertConfNag2SantokuCsv.sh is a Bash script converting this CSV file from the ConfNag to the Santoku format. You can safely forget about this, unless you're eager to know what a mess it was !
  • data.csv : your CSV file. You can name it as you like. Don't forget to declare it in modules/config.py.
  • config/GENERIC/*.ini : Santoku works by searching/replacing values in patterns. These files are the generic patterns, all Santoku users need them.
  • config/plugins/*.ini : These are the plugins patterns. Depending on the plugins you use, you will have to build your own patterns. Patterns provided here must be considered as examples only : some are based on custom-made plugins (not (yet?) released to the community), others are totally fake and were made for development/testing purposes only.
  • doc/documentation.html : congratulations : you're just RTFM !!!
  • modules/config.py : main configuration file. Many (MANY !) values are defined there to avoid hardcoding. So even though it looks like it's highly configurable, tampering options may have undesired effects. The options you can safely edit here are csvFileName, outputPath, outputFileHosts, outputFileServices, outputFileCommands. However, since Santoku is free software, you are strongly encouraged to have a look at the code, play with it, and improve it !
  • modules/*.py : these are the Santoku internals. Feel free to have a look at it, and let me know anything that could be improved !

Usage

How to get Santoku ?

git clone https://github.com/Httqm/Santoku.git

This will create a Santoku sub-directory in the current directory.

Installation
  1. copy deployToShinken.sh-dist into deployToShinken.sh. This will prevent you from overwriting your local settings if you git pull for Santoku updates.
  2. copy modules/config.py-dist into modules/config.py. Skipping this step will cause the error : 
    File "[your installation path]/Santoku/modules/debug.py", line 22, in <module>
	from modules import config
ImportError: cannot import name config
    since the expected file modules/config.py can not be imported by modules/debug.py as it doesn't exist.
  3. you can move the Santoku directory around to any desired location.

For the impatients : the 'cp' operations above can be performed by executing the RUN_ME_AFTER_GIT_CLONE.sh script.

Running Santoku

Santoku has no command-line flag since every option it requires is stored in its configuration file modules/config.py.
To run Santoku, just type ./santoku.py

But before running it, you'll have to :

  • Make backup copies of the hosts.cfg, services.cfg and commands.cfg files as they will be overwritten by versions created by Santoku.
  • Define hosts and services in the CSV file
  • Setup a .ini file for each plugin you use
Format of the CSV file

Some of the columns of the CSV file are generic, then mandatory. Others are used to define the services, and depend on your setup :

Generic columns :
  • host_name : as defined in the Nagios documentation.
  • alias : as defined in the Nagios documentation.
  • hostgroups : "|"-separated list of groups this host belongs to. Example : groupA|groupB|groupOther.
  • address : as defined in the Nagios documentation.
  • check_command : as defined in the Nagios documentation.
  • ignore_host : boolean. Put a 1 in the corresponding cell to completely ignore a host.
  • parents : as defined in the Nagios documentation.
  • hostDirectivesNames : "|"-separated list of Nagios host definition directives. Example : notifications_enabled|notification_interval.
  • hostDirectivesValues : "|"-separated list of values for the directives declared in the previous column. Example : 1|120. This example means we have : notifications_enabled = 1 and notification_interval = 120.
  • sshLogin : The login to use for SSH commands to be executed on the host defined by host_name.
  • END : A final column is mandatory, as data is expected between CSV separators (, or ; depending on the software used to make the CSV export). Without this final column, the value of the previous column would have a CSV separator on the left, but not on the right, and would be ignored.
Plugin-specific columns :

These columns are named following a serviceName:columnName pattern. These columns are repeated for every service.

  • service1Name:do : boolean. A 1 enables this service for the current host (the host defined on the same line of the CSV file). Anything else disables it.
  • service1Name:warn : service warning threshold.
  • service1Name:crit : service critical threshold.
  • service1Name:description : service description.
  • service1Name:serviceDirectiveNames : "|"-separated list of Nagios service definition directives. Example : check_interval|notifications_enabled.
  • service1Name:serviceDirectiveValues "|"-separated list of values for the directives declared in the previous column. Example : 1|0. This example means we have : check_interval = 1 and notification_enabled = 0.

One of the cool features of Santoku is that it allows providing parameters as a list in the CSV file. For instance, if you want to monitor /, /home and /var on a single host, you needn't specifying 3 services. You just have to declare the targets as a "|"-separated list in the corresponding column (here : check_disk:disk) : /|/var|/home. Thus, all 3 services will share the same warning and critical thresholds.

But if you prefer specifying different thresholds for the 3 services, it's still possible by providing a list of warning thresholds : 25|26|27 OR a list of critical thresholds : 10|11|12, or both.

Known bugs and limitations

Since Santoku is in its early version, few bugs and limitations are known so far. Here are some important notices :

  • OBSOLETE ! In the commands.cfg file, Santoku writes the host checks and service checks commands. In a Nagios / Shinken fresh install, the commands.cfg file not only stores these commands, but also the notification commands. So overwriting the default commands.cfg will erase the notification commands. To fix this :
    1. create a commands_notification.cfg file in the directory containing the commands.cfg file on the host running Shinken
    2. save the notification commands notify-host-by-email and notify-service-by-email in commands_notification.cfg
    3. register commands_notification.cfg in nagios.cfg by declaring it like commands.cfg : cfg_file = commands_notification.cfg
  • Santoku is designed so that, upon error, it exits cleanly with a detailed error message. If it happens to die on a Python error : this is a BUG ! Please report an issue.
